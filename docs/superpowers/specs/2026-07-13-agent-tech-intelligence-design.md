# Agent 技术情报推送系统设计

## 目标

构建一个由 GitHub Actions 托管的个人技术情报服务：每日向指定 QQ 邮箱发送 Agent 开发内容摘要，并在每周生成一份趋势周报。系统将完整 Markdown 报告保存于私有 GitHub 仓库，供长期检索和复盘。

## 范围

第一版覆盖以下内容，按优先级排序：

1. GitHub 热门 Agent 项目。
2. Agent 相关论文与工程实践文章。
3. Agent 框架、模型与平台的官方更新。

系统每天生成日报、发送邮件并保存报告；每周日额外生成和发送周报。第一版不调用付费 LLM API，使用确定性的规则和模板生成中文摘要，以保持零运行成本。

不在第一版范围内：网页前端、数据库、用户管理、多收件人支持、实时推送、付费模型生成长文解读。

## 架构

Python 命令行应用由 GitHub Actions 的两个定时工作流调用。采集器各自负责一种来源；统一的内容管道负责归一化、相关性筛选、去重、排序和报告生成。邮件模块只接收已生成报告，负责通过 QQ 邮箱 SMTP 投递精简摘要。

```text
GitHub Actions（每日 / 每周 / 手动）
  -> 采集器（GitHub、论文与工程文章、官方更新）
  -> 归一化、相关性过滤、去重、优先级排序
  -> Markdown 日报 / 周报
  -> Git 提交 reports/ 历史文件
  -> QQ SMTP 邮件摘要
```

模块之间以稳定的数据对象传递内容；网络请求、报告渲染、邮件投递相互隔离，以便单独测试和替换。

## 内容来源与选择规则

### GitHub 项目

通过 GitHub REST API 搜索近期开源仓库，使用 `agent`、`agentic`、`multi-agent`、`MCP`、`tool-use` 等关键词。记录仓库名称、描述、URL、语言、许可证、star 数、创建或更新时间。按近期热度、关键词相关性和更新鲜度排序。

### 论文与工程实践

通过公开 RSS 或 API 采集 arXiv、Hacker News 和预先配置的工程博客。仅保留标题或摘要与 Agent 开发直接相关的内容；记录来源、发布时间、URL 和简短中文要点。

### 官方更新

通过配置化 RSS/发布页面抓取 OpenAI、Anthropic、Google、LangChain、LlamaIndex、CrewAI、Microsoft AutoGen 等官方更新。仅保留产品、API、框架和工程实践的更新。

所有来源都必须返回原始链接；抓取失败的单个来源不会阻止其他来源产出报告。

## 数据处理

所有采集结果归一化为包含 `title`、`url`、`source`、`category`、`published_at`、`summary` 和 `score` 的条目。

- 以规范化 URL 为主键去重；标题高度相似且来自同一项目/文章时合并。
- 过滤非 Agent 开发主题、缺少有效 URL、超过时效窗口或重复的条目。
- 报告按“GitHub 项目、论文与工程实践、官方更新”分区；同一区内按分数和时间倒序排列。
- 日报默认最多 15 条，优先保留 GitHub 项目；无新内容时仍生成状态日报并发送“无新增”的健康状态邮件。
- 中文摘要使用标题、描述和预设模板生成，明确链接回原始来源，不伪造未采集到的事实。

## 报告与邮件

日报路径为 `reports/daily/YYYY-MM-DD.md`。邮件主题为 `Agent 技术日报 · YYYY-MM-DD`，正文包含最多 15 条简短要点及对应报告/来源链接。

周报于每周日生成，路径为 `reports/weekly/YYYY-Www.md`。周报读取前七天日报，包含：本周新增总量、按类别统计、最值得关注项目、论文与工程实践清单、官方更新清单和可追溯链接。邮件主题为 `Agent 技术周报 · YYYY-Www`。

报告会由工作流提交回私有仓库。运行日志与失败信息保留在 GitHub Actions 中；报告不包含 SMTP 授权码或任何 Secret。

## 自动化与时区

每日工作流在北京时间 08:30 触发（cron 使用 UTC 00:30）；每周工作流在周日北京时间 09:00 触发。两个工作流均支持 `workflow_dispatch` 手动运行。

GitHub Actions 定时任务可能延迟，系统以运行日期生成报告并保证重复执行幂等：同一天或同一周再次运行时更新对应报告，而不是创建重复文件。

## 安全与配置

非敏感配置通过版本控制的示例配置文件提供，实际值可由环境变量覆盖。以下值仅保存为 GitHub Actions Secrets：

- `QQ_SMTP_USER`：QQ 发件邮箱。
- `QQ_SMTP_AUTH_CODE`：QQ 邮箱 SMTP 授权码。
- `EMAIL_TO`：收件邮箱。
- `GITHUB_TOKEN`：用于 GitHub API 认证和提交报告的令牌。

工作流遵循最小权限原则，只启用读取内容和写入本仓库报告所需的权限。程序输出不得打印 Secret、授权码或 SMTP 登录命令。

## 失败处理与可观测性

单个来源超时、返回无效数据或请求失败时，记录来源级错误并继续处理其余来源。若没有任何来源成功，任务失败且不发送“无新增”邮件；GitHub Actions 将保留失败日志，方便手动重试。

报告生成成功但邮件投递失败时，日报/周报仍提交到仓库，工作流标记为失败以便发现和重试。邮件失败不删除报告。

## 测试策略

单元测试覆盖：条目归一化、URL 去重、主题筛选、排序与限额、日报渲染、周报汇总、邮件消息构造和配置验证。网络请求与 SMTP 传输通过可注入接口替换为测试替身；测试不访问真实 GitHub、RSS 或 QQ SMTP。

集成级测试使用固定夹具模拟三类来源，验证端到端生成 Markdown 报告。GitHub Actions 在每次代码改动时运行测试，定时工作流仅在测试通过的主分支版本上运行。

## 验收标准

1. 手动触发日报工作流可生成并提交一份 Markdown 日报，且向配置收件人发出 QQ 邮件。
2. 日报涵盖三类来源，并按指定优先级显示；同一链接不会重复出现。
3. 手动触发周报工作流可从最近七份日报生成周报并发邮件。
4. 单一来源失败不会阻塞其他来源、报告或邮件；所有来源失败和邮件失败均使工作流明确失败。
5. 仓库与报告中不出现 SMTP 授权码、访问令牌或其他 Secret。
6. 自动化测试覆盖核心处理与渲染逻辑，并在 GitHub Actions 中执行。
